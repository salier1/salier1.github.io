Computer Prototype

A digital walkthrough of the gaze-aware streaming experience with supporting documentation for evaluators and participants.

Prototype Overview

System Description

The current build, the “Gaze Interaction Prototype Suite,” packages three gaze-inspired interaction concepts into a single HTML experience that stakeholders can launch without additional tooling.

Launch the computer prototype in a new tab

  • Gaze-Adaptive Player. A focus spotlight trails the viewer’s point of regard, sharpening the focal region while softening the periphery.
  • Attention Heatmap Dashboard. A timed tracking task logs cursor or gaze events, awards points for hits, and renders a cumulative heatmap overlay.
  • Adaptive Bandwidth Controller. Facilitators can simulate network conditions, toggle adaptive streaming, and inspect bandwidth usage alongside availability.

Key characteristics:

  • Self-contained in index.html with no build or backend step.
  • Right-side drawer centralises toggles, sliders, and diagnostics.
  • Hero landing screen doubles as onboarding copy and collapses to maximise the stage.
  • Accessibility notes include keyboard-accessible controls, ARIA labels, and stateful badges for adaptive modes.

User Goals & Scenarios

  • Faculty presenters demonstrate adaptive streaming to prospective partners, quickly enabling or disabling spotlight and bandwidth features during live walkthroughs.
  • HCI researchers capture gaze/cursor events while participants complete timed tracking tasks, exporting heatmap states for follow-up analysis.
  • Technical facilitators stress-test network states in the same interface, correlating playback quality with simulated bandwidth dips.

Each scenario starts with the hero overview, allowing the facilitator to brief newcomers before collapsing the panel and focusing on the active module. The consistent drawer layout reinforces muscle memory so users do not have to hunt for controls as they switch context.

Current Evaluation Focus

Recent usability sessions examine how quickly observers identify the spotlight’s effect, whether the heatmap hints aid situational awareness, and how intuitively facilitators notice bandwidth bottlenecks. Metrics include time-to-first-success for each task, perceived clarity of status badges, and the number of prompts required to locate key toggles.

These insights feed the iteration backlog—for example, tweaking badge contrast, refining tooltip copy, and smoothing the transition when the hero panel collapses into the top navigation strip.

Design Evolution (with rationale)

No. Change Summary Why We Changed It Usability / Feasibility Rationale
1 Merged multi-page flow into a single entry point and surfaced the spotlight toggle in Task 1. Feedback: Low-fidelity testers were unsure which HTML file to start with and asked for “single file, single toggle” to ease onboarding. A unified index.html removes file confusion and places the enable spotlight control where viewers expect it, reducing setup time.
2 Added hero task overview with collapsible cards via a Hide task overview toggle. Feedback: Early demos valued the overview cards but wanted to reclaim screen space during presentations. The toggle preserves onboarding context for first-time viewers while allowing presenters to clear the viewport instantly.
3 Introduced a unified drawer with integrated diagnostics covering spotlight settings, heatmap workflow, and bandwidth monitoring. Feedback: Reviewers requested consistent access to controls without swapping files or scrolling long pages. Centralising controls improves discoverability, reduces cognitive load, and enabled the bandwidth monitor toggle.

Supporting Iterations

  • Bandwidth monitor toggle and status badge to declutter the viewport and surface automation states only when needed.
  • Task HUD polish that keeps score and timer feedback persistent during the heatmap exercise.
  • Badge repositioning so the adaptive encoding badge no longer collides with the focus settings button.

Collectively, these changes resolve the entry-point confusion, blocked stage space, and scattered controls highlighted in low-fidelity testing, resulting in a faster, more intuitive walkthrough.

4. Usability Test Plan

4.1 User Population

Attribute Definition
Target users Knowledge workers and graduate students who routinely consume remote presentations, training clips, or screen-share walkthroughs.
Sample size 6–8 participants per round.
Background Comfortable with modern browsers, mouse/trackpad navigation, and multitasking between content and controls.
Usage context Adaptive media on laptop/desktop monitors in quiet office or home settings, plus occasional shared screens during workshops.

Recruitment: Use campus mailing lists and HCI lab volunteer pools that match the above profile.

4.2 Usability Goals

ID Goal Statement Quantitative Target
UQ1 — Learnability Users can enable or disable adaptive controls (focus spotlight, heatmap, bandwidth monitor) without facilitator guidance. ≥90% locate and activate the requested control within 30 s; first complete task run finishes in ≤2 min without help.
UQ2 — Efficiency Adaptive aids reduce effort to acquire targets and complete tracking tasks. Adaptive ON target acquisition time ≤80% of adaptive OFF baseline; cursor travel per hit reduced by ≥15%.
UQ3 — Satisfaction Adaptive visuals feel comfortable and pleasant. ≥80% rate comfort, clarity, and visual pleasantness ≥4/5 on post-task Likert scales.
UQ4 — Perceived Clarity Adaptive blur enhances focus on key regions. ≥75% report improved clarity with adaptive ON; heatmap density on target zones increases by ≥25%.
UQ5 — Robustness Adaptive streaming remains stable under network fluctuation. Update latency <80 ms; ≥95% of simulated runs show no flicker or contrast loss.
UQ6 — Awareness & Control Users feel in control and not distracted by automation. ≥80% report feeling in control; <10% describe adaptive effects as distracting or confusing.

4.3 Test Procedure

  1. Staffing: Two examiners per session — a facilitator to run the script and a note-taker to capture metrics and observations.
  2. Environment & equipment: 15″–24″ laptop/desktop, external mouse, latest Chrome/Edge, stopwatch, consent forms, pre/post surveys, and optional screen/audio capture (with consent).
  3. Prototype handling: Load index.html (direct open or Live Server), referencing the user manual to confirm spotlight, heatmap, and bandwidth controls before the participant arrives.
  4. Participant briefing: Explain study purpose, collect demographics, remind participants they can withdraw, and encourage think-aloud behaviour.
  5. Task script (≈25 min):
    1. Warm-up — Toggle focus spotlight, adjust radius.
    2. Task 1 — Locate spotlight toggle and resize spotlight.
    3. Task 2 — Run a 45 s heatmap session, achieve ≥5 hits, review the overlay.
    4. Task 3 — Enable adaptive bandwidth, simulate “Limited” and “Boosted” networks, interpret the bandwidth monitor.
    Facilitator only offers hints after 90 s of stall time and records assistance.
  6. Interaction rules: Facilitator clarifies goals but avoids step-by-step coaching; respond to “Where is X?” with neutral prompts.
  7. Data capture: Track time-on-task, errors, assistance, cursor metrics (optionally via heatmap export), and gather post-task Likert ratings plus qualitative notes.
  8. Post-session wrap-up: Administer SUS-lite or custom survey, run a brief interview, thank the participant, and anonymise stored data.

4.4 Reporting

Data Recording & Reporting Workflow

Step Details
Capture Use the shared Google Sheet for timing, errors, assistance, and heatmap exports; store media captures in the secure Drive folder.
Survey aggregation Google Forms summarises Likert scores; export CSV for deeper analysis.
Synthesis Weekly debriefs cluster issues, track goal attainment, and update the backlog.
Deliverables Produce a 2–3 page PDF summarising metrics, findings, priority issues, and recommended actions; upload to the course portal and archive in /reports/usability-round1.pdf.

Evaluation Methods Overview

Evaluation Method Metrics & Justification Application
Lab-based task testing Completion time, error counts, assistance — map directly to UQ1/UQ2 for learnability and efficiency. Measure during scripted tasks; compare adaptive ON vs. OFF runs.
Perceived clarity & satisfaction survey Likert ratings on comfort, clarity, and control support UQ3/UQ4/UQ6. Post-task survey plus short interview for qualitative depth.
Heatmap / cursor analytics Fixation density and cursor distance reinforce UQ2 and UQ4 targets. Export built-in heatmap arrays or capture via observation to quantify focus behaviour.
Network stability test Latency logs and visual artefact checks validate UQ5 robustness. Monitor during Task 3 simulations, using console timestamps and observer notes.
Heuristic evaluation (Nielsen 10) Focus on visibility of system status, user control, and real-world match. Conducted by two HCI evaluators before and after user testing to surface structural issues early.

Combining quantitative sessions with expert heuristic reviews ensures the adaptive prototype evolves based on measurable outcomes and qualitative insight.
